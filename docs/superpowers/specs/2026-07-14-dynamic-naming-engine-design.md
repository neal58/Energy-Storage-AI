# 动态中文宝宝取名引擎设计

## 1. 最终决策

项目不再采用“固定完整名字库中挑选”的方案，也不采用“输入八字后按缺什么补什么”的传统模板方案。

第一版采用 **规则引擎 + 汉字知识库 + 动态组合生成 + 多阶段审查** 的混合架构：

1. 父母姓氏、孩子采用的姓氏、出生日期时间、出生地时区、性别、风格、寓意、辈分字和避讳字共同形成一次取名任务。
2. 系统根据本次任务动态筛选汉字、选择语义结构并组合候选，而不是返回预先写好的完整名字。
3. 出生信息真实参与候选生成，但只提供历法、节气、时辰和传统五行出现情况等可解释参考，不宣称能决定命运。
4. 所有候选必须经过全名音律、语义完整性、现实姓名自然度、书写使用、热门程度和谐音风险审查。
5. 最终只展示少量高质量结果，并明确说明每个名字为什么适合本次输入。

## 2. 产品目标

用户填写同一组偏好，但改变父姓、母姓、出生时间或孩子姓氏后，候选名字应发生有依据的变化。

产品必须做到：

- 同一个名字不会对所有姓氏给出相同评价。
- 出生时间不是摆设，节气、时辰和传统参考会改变候选字权重。
- 母亲姓氏可以通过直接随姓、双姓、意象纪念或不参与四种方式进入生成逻辑。
- 不使用生僻字制造“小众感”。
- 不编造诗词出处、不伪造重名率、不输出无法解释的“98分”。
- 推荐结果像现实中的正式姓名，而不是小说名、网名或两个好字的随机拼接。

## 3. 第一版范围

### 3.1 用户输入

必填：

- 父亲姓氏
- 母亲姓氏
- 孩子姓氏方式：随父姓、随母姓、父母双姓
- 性别倾向：男孩、女孩、中性
- 出生日期和当地时间
- 出生地时区

选填：

- 出生城市或经度，仅用于展示和后续真太阳时扩展
- 是否启用传统历法参考
- 是否让母姓参与名字：不参与、意象纪念、直接作为名字首字
- 名字长度：单字名、双字名
- 风格：温润、清朗、稳重、书卷、现代简洁、自然、古典
- 核心寓意：智慧、品格、志向、平安、光明、成长、从容、责任
- 辈分字或指定字
- 避开字
- 是否降低近年高频字权重
- 用户自己已知的不良谐音或家庭避讳词

### 3.2 不在第一版实现

- 自动判断“喜用神”。不同流派算法差异较大，第一版只展示四柱与五行出现情况，并作为弱权重参考。
- 通过姓名预测性格、财富、健康、婚姻或命运。
- 大规模账号系统、付费功能和云端训练。
- 完整方言语音库。第一版进行普通话审查，并让用户录入家庭方言避讳词。
- 依赖大量真实用户反馈才能运行的机器学习模型。

## 4. 事实与传统参考

### 4.1 历法计算

使用 `lunar-javascript` 进行公历、农历、节气、干支和四柱计算。库版本固定，不使用浮动的 `latest`。

公历与农历信息以出生地当地民用时间作为输入。页面明确展示：

- 当地出生时间与时区
- 农历日期
- 当前节气或相邻节气
- 年柱、月柱、日柱、时柱
- 五行在四柱中的出现次数

八字的年柱以立春交接时刻划分，月柱以节令交接时刻划分；晚子时存在流派差异。第一版采用库的默认流派 2，并在出生时间为 23:00—00:59 时显示边界提示。

### 4.2 出生信息如何参与取名

出生信息转化为三类软标签：

1. **季节标签**：春生、生长、和煦；夏生、明朗、充沛；秋生、澄澈、收获；冬生、沉静、坚韧。
2. **时辰标签**：晨间偏向初、明、晞、朝等意象；日间偏向朗、昭、景；傍晚偏向宁、晏、和；夜间偏向静、安、澄。
3. **传统五行标签**：统计木、火、土、金、水出现情况，仅对相关语义标签提供小幅权重，不执行“缺水就用涵、缺木就用梓”的机械补字。

出生信息在总排序中的影响上限为 15%，不能压倒音律、语义和现实使用性。

## 5. 父母姓氏参与方式

### 5.1 孩子姓氏

- 随父姓：完整姓名使用父姓。
- 随母姓：完整姓名使用母姓。
- 双姓：父姓在前、母姓在后，作为复姓处理；默认生成单字名，避免四字姓名过长，用户可选择双字名。

### 5.2 母姓纪念

当孩子不随母姓时，提供三种方式：

- 不参与：母姓只保存在任务信息中。
- 意象纪念：将母姓映射为克制的语义标签。例如林映射为生长、清润；江映射为开阔、流动；周映射为周全、秩序。映射必须有人工维护，不存在可靠映射时不强行参与。
- 直接入名：母姓作为名字第一个字。此模式必须经过全名自然度和谐音审查，低质量时允许返回“本条件下不建议直接入名”。

## 6. 动态知识库

### 6.1 汉字事实库 `characters`

每个汉字包含：

- 汉字
- 拼音、声调、声母、韵母
- 是否多音字
- 常用度
- 笔画与书写难度
- 性别倾向
- 现代语义标签
- 风格标签
- 传统五行标签及来源类型
- 季节和时辰意象标签
- 适合在名字第一字、第二字或单字名中的角色
- 热门字等级
- 风险标签

第一版目标为 300—500 个常用、易认、适合姓名使用的汉字。完整名字不作为主要数据单元。

### 6.2 语义关系库 `relations`

记录两个字为什么可以组合，而不是仅记录“都寓意好”。

关系类型：

- 修饰 + 核心：嘉言、清和
- 认知 + 方向：知远、思齐
- 行为 + 原则：守正、修远
- 状态 + 品格：怀谦、明哲
- 自然意象 + 人格寓意
- 家庭纪念 + 核心寓意

每条关系包含兼容度、语义说明模板和禁配条件。没有关系边的组合默认不得进入最终候选。

### 6.3 结构模板 `templates`

模板决定两个字的语法和语义角色，例如：

- `quality + virtue`
- `mind + direction`
- `action + principle`
- `scene + clarity`
- `growth + stability`
- `maternal_symbol + virtue`

模板不是完整名字列表。同一模板根据本次输入可生成不同组合。

### 6.4 姓氏音韵库 `surnames`

常见姓氏记录：

- 拼音、声调、声母、韵母
- 常见谐音风险
- 不宜紧接的同音或近音字
- 单姓和复姓长度规则

未知姓氏允许用户手动确认拼音；未确认时只执行文字和通用声调审查。

## 7. 候选生成流程

### 阶段 A：任务建模

将用户输入整理为 `NamingTask`：

```text
surname
fatherSurname
motherSurname
surnameMode
gender
birthProfile
styleWeights
meaningWeights
requiredChars
avoidChars
maternalMode
trendPreference
customTaboos
```

### 阶段 B：候选字池

从汉字事实库中进行硬过滤：

- 避开字、家庭避讳字
- 高风险多音字
- 极低常用度字
- 书写过度复杂字
- 与性别和名字位置严重冲突的字

再根据风格、寓意、出生标签、母姓纪念和热门程度进行软排序，得到第一字池和第二字池。

### 阶段 C：动态组合

按结构模板和语义关系边生成 300—3000 个候选组合：

- 指定字存在时固定对应位置或同时尝试两个位置。
- 双姓默认优先单字名。
- 母姓直接入名时固定第一字。
- 使用任务种子保证同一输入首轮可复现，“换一批”只改变采样种子。

### 阶段 D：硬审查

候选必须通过：

- 全名重复字检查
- 姓与名首字同音、近音检查
- 连续声母或韵母过度重复检查
- 三个音节完全同调检查
- 普通话不良谐音词表
- 用户自定义避讳词
- 明显品牌名、网名和小说化组合规则
- 无语义关系边或解释不成立的组合

### 阶段 E：多维排序

权重固定为：

- 现实姓名自然度 25%
- 全名音律与发音 20%
- 语义完整性 20%
- 姓氏适配 10%
- 成长与正式场景适配 10%
- 用户风格与寓意 10%
- 出生与家庭纪念参考 5%

传统五行和时辰仅包含在最后 5% 中；加上季节意象后，所有出生相关因素总影响不得超过 15%。

### 阶段 F：多样化选取

最终选取 8 个，不按纯总分截取，避免一批结果全部相似：

- 综合最贴合
- 稳妥耐用
- 小众但自然
- 父母纪念方向
- 出生意象方向
- 现代简洁
- 书卷文化
- 探索方向

若某一类没有合格候选，则减少结果数量，不用低质量名字凑数。

## 8. 输出解释

每张名字卡必须显示：

- 完整姓名与拼音
- 推荐方向
- 名字完整语义，而不是逐字堆砌
- 与姓氏连读的结论
- 哪些用户偏好影响了该名字
- 出生信息的实际贡献，例如“出生于秋季，系统提高了澄澈、收获类意象的权重”
- 母姓是否、如何参与
- 可靠出处；没有直接出处时明确说明
- 风险提示和需要家庭复核的事项

不显示虚假的精确重名率和不可验证的吉凶分数。内部排序分数只用于排序，前端使用“强推荐、值得考虑、风格探索”三级结论。

## 9. 用户反馈

用户可对单个候选选择：

- 喜欢这个方向
- 太普通
- 太生僻
- 太古风
- 太柔
- 太硬
- 不喜欢某个字
- 读起来不好

反馈即时修改当前任务权重并重新生成，不依赖外部用户训练。页面刷新前保存在 `localStorage`；不上传个人出生信息。

## 10. 技术架构

项目仍为静态 Web 应用，但源代码模块化，并构建为一个可直接预览的单文件 HTML。

- `lunar-javascript@1.7.7`：历法、节气、四柱和五行展示。
- ES Modules：源代码开发。
- `esbuild`：将应用和依赖打包。
- 自定义构建脚本：将 CSS 和 JS 内联到根目录 `index.html`，避免 HTML Preview 无法加载模块的问题。
- Node 内置测试运行器：逻辑测试。
- GitHub Actions：执行测试和构建，只有全部通过才合并。

核心模块：

```text
src/calendar/birth-profile.js
src/data/characters.js
src/data/relations.js
src/data/templates.js
src/data/surnames.js
src/data/maternal-symbols.js
src/data/risk-lexicon.js
src/engine/task.js
src/engine/character-pool.js
src/engine/generate.js
src/engine/phonology.js
src/engine/semantics.js
src/engine/risk.js
src/engine/rank.js
src/engine/explain.js
src/app.js
src/styles.css
scripts/build-single-html.mjs
```

## 11. 错误处理

- 出生时间缺失：允许关闭传统参考继续取名，但明确提示出生信息未参与。
- 出生地时区缺失：阻止启用传统参考，要求选择时区。
- 指定字导致无结果：解释是哪条硬规则淘汰了候选，并允许用户放宽指定字位置。
- 母姓直接入名无自然结果：明确建议改用意象纪念，不强行输出。
- 历法库加载或计算失败：降级到季节和当地时段标签，不伪造四柱。
- 未知姓氏拼音：提示用户确认拼音；在确认前降低音律结论置信度。

## 12. 验收标准

1. 改变父姓或孩子姓氏方式，完整姓名、音律排序和至少一半候选发生变化。
2. 保持其他输入不变，将出生时间从清晨改为夜间，出生意象路线和部分候选发生变化。
3. 改变出生月份跨越季节，季节标签及部分候选发生变化。
4. 母姓选择“不参与”和“意象纪念”时，解释和候选集合不同。
5. 指定字必须出现在所有结果中；避开字不得出现。
6. 同一输入和同一批次种子结果可复现；点击“换一批”产生新的合格候选。
7. 不出现无语义关系支撑的随机组合。
8. 不输出伪造出处、伪造重名率和命运断言。
9. 自动测试和构建均通过。
10. 根目录 `index.html` 在 HTML Preview 中能够完成输入、生成、换一批和反馈操作。

## 13. 依据与限制

- 农历计算参照现行国家标准 GB/T 33661-2017《农历的编算和颁行》。
- 历法实现采用开源 `lunar-javascript`，其文档明确提供节气、八字及各柱五行接口。
- 中文宝宝命名属于多目标短文本生成问题，应采用“需求提取—候选生成—多目标评价—迭代优化”，而不是单次随机生成。
- 传统八字与五行部分作为文化参考展示，产品不把它包装为科学预测。
