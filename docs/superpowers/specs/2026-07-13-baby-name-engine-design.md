# 宝宝取名引擎设计

## 目标
构建一个以“自然、顺口、耐听”为第一优先级的中文宝宝取名工具。产品不再随机拼接单字，而是通过高质量候选生成、语言规则过滤、文化证据校验和用户反馈排序，输出可解释、可比较、可持续学习的名字方案。

## 产品原则
1. 好听优先，出处加分，不为出处牺牲自然度。
2. 综合平衡推荐：大众耐听、小众高级、传统文化三类并存。
3. 传统命理仅作可选参考，不作为核心排序依据。
4. 评分必须可解释，区分事实、数据与审美判断。
5. 第一轮只给 6 个名字，用户反馈后再生成下一轮。

## 第一阶段范围
- 输入：姓氏、性别倾向、名字长度、风格偏好、寓意偏好、避开字、指定字。
- 输出：6 个候选名字，分为最贴合、稳妥型、小众型、文化型、现代型、探索型。
- 反馈：喜欢、太普通、太古风、不好听、字复杂、寓意不喜欢。
- 再推荐：根据反馈调整下一轮排序与候选池。
- 对比：最多选择 3 个名字进行音律、寓意、出处、独特性、书写、成长适配对比。

## 核心架构
### 1. 候选生成器
候选来自三类来源：
- 经过审核的优质名字组合库。
- 公版古籍与诗词中经过语境审核的词组。
- 受约束的模型生成候选，必须经过后续过滤与评分。

### 2. 硬过滤器
过滤多音字高风险、生僻字、明显谐音、姓氏连读歧义、重复声韵、字义冲突、像品牌或网名的组合。

### 3. 审美评分器
主要维度：
- 全名自然度 30%
- 音律与发音流畅度 25%
- 语义完整度 15%
- 姓氏适配度 10%
- 成长适配度 10%
- 独特性与书写友好度 10%

文化出处单列展示，不直接把总分抬高到压倒“好听”。

### 4. 反馈学习器
用户反馈转化为偏好向量，例如：
- 喜欢“清和”提升清雅、简洁、耐听权重。
- “太普通”提升独特性权重。
- “太古风”降低古典词组权重。
- “不好听”降低同类音节与声调结构。

第一阶段使用本地会话学习；接入云数据库后保存匿名聚合反馈。

## 数据结构
### characters
汉字、拼音、声调、多音字标记、笔画、常用度、性别倾向、语义标签、风险标签。

### name_candidates
名字、拼音、声调模式、风格、寓意、来源类型、出处文本、人工审核等级、热度、书写难度。

### surname_rules
姓氏、声调、常见谐音、建议声调模式、禁止组合。

### feedback_events
匿名会话、姓氏、候选名、反馈类型、时间、偏好向量变化。

## 页面流程
1. 基础信息。
2. 3 到 5 组审美偏好选择。
3. 首轮 6 个候选。
4. 反馈后再推荐。
5. 3 个名字对比。

## 错误与风险处理
- 没有足够高质量候选时，减少数量，不用低质量结果凑满。
- 出处无法确认时明确标记“无可靠出处”，不得编造。
- 重名与流行度没有权威数据时只显示“暂无可靠统计”。
- 方言谐音无法覆盖时提示用户进行家庭方言复核。

## 测试要求
- 姓氏与名字连读规则测试。
- 多音字、生僻字、禁用字过滤测试。
- 反馈后排序发生预期变化的测试。
- 无可靠出处时不得生成虚假出处的测试。
- 移动端核心流程可用性测试。

## 第一阶段不做
- 八字喜用神自动判定。
- 付费会员。
- 大规模用户账号体系。
- 宣称拥有百万真实姓名数据。
